# [설계 문서] 태스크 저항 예측 기반 선제개입 시스템

> 분류: 현재는 영업비밀로 관리. 특허 출원 여부는 별도 검토 (요소 조합에 따라 추후 특허 포인트 가능성 있음) — 외부 공유 금지
> 관련 문서: [특허성_검토_보고서.md](./특허성_검토_보고서.md) (번아웃 A/B/C 로직, 이건 별도로 특허 출원 트랙)
> 구현 상태: 1~4일차(로드맵 9장) 구현 완료. 이후 LLM 태그 분류 방식을 걷어내고 habitId/텍스트 기반 결정적 그룹핑으로 리팩터링함 (아래 3·4장이 최신 구조).

---

## 1. 목적

기존 번아웃 개입 로직(streak/완료율 기반, `chat_screen.dart` `_maybeOfferRest`)과 이월 기반 확인 로직(`coach_config.dart`, "미룬 횟수")은 둘 다 **완료 여부**를 기준으로 반응한다. 이 시스템은 완료 여부와 독립적인 **정서적 저항 신호**를 추적해, 사용자가 불만을 표출하기 *전에* 코치가 먼저 말을 걸도록 하는 걸 목표로 한다.

핵심 타겟: "결국 하긴 하지만 매번 투정부리는 일" — 기존 시스템이 놓치는 영역.

---

## 2. 전체 파이프라인

```
[신호 수집] → [그룹핑(habitId/텍스트)] → [저항점수 계산] → [개입 판단] → [쿨다운 체크] → [프롬프트 지침 주입] → [결과 관찰] → [가중치 갱신]
                                        ↑___________________________________________________________________________|
                                                              (폐루프)
```

## 3. 데이터 모델

기존 `TaskItem`(id, text, category, done, habitId 등)에 없는 필드가 필요하다. 별도 저장소로 분리하고 `taskId`로 연결한다 (기존 TaskItem 구조를 건드리지 않기 위함).

### 3.0 그룹핑 방식 — LLM 태그 분류를 걷어내고 결정적 계산으로 대체

**처음 설계는 LLM에게 "이 태스크에 어울리는 태그 3개를 골라줘"라고 요청해서, 텍스트가 다른 태스크끼리도 공통 원인(태그)으로 묶는 방식이었다.** 예: "논문 쓰기"와 "대청소"가 둘 다 "장시간" 태그를 공유하면 같은 저항 원인으로 취급.

**재검토 후 이 방식을 폐기했다.** 이유:
- 실제로 사용자가 반복해서 저항하는 일은 대부분 **완전히 새로운 일회성 일이 아니라 습관·일정처럼 반복되는 일**이다. 이런 반복 항목은 이미 앱 데이터에 `habitId`라는 정확한 식별자가 있어서, LLM 분류 없이도 "이 습관, 저번에도 저항 있었네"를 정확하게 알 수 있다.
- LLM 태그 분류는 (1) 호출마다 결과가 미묘하게 달라질 수 있고(같은 "방 치우기"가 "정리정돈"으로도 "집안일"로도 분류될 수 있음), (2) "완전히 새로운 일회성 태스크가 과거의 다른 태스크와 의미상 비슷해서 묶이는" 좁은 케이스 하나를 잡자고 분류 인프라(프롬프트, 활성 태그 5개 관리, 재분류 방지 로직)를 통째로 유지해야 했다. 배보다 배꼽이 큰 판단.

**그래서 그룹핑 키(`groupId`)를 결정적으로 계산한다. 우선순위는 habitId → 사전 정의 카테고리 키워드 → 정규화 텍스트 순이다:**

```dart
const categoryKeywords = {
  '집안일': ['청소', '설거지', '빨래', '정리', '분리수거', '화장실', '먼지'],
  '운동': ['운동', '헬스', '조깅', '요가', '스트레칭', '산책'],
  '공부': ['공부', '독서', '강의', '시험', '과제'],
  '콘텐츠 제작': ['SNS', '숏츠', '영상', '카드뉴스', '포스팅', '블로그', '릴스', '유튜브'],
};

String computeGroupId({String? habitId, required String taskText}) {
  if (habitId != null && habitId.isNotEmpty) return 'habit_$habitId';
  final category = matchCategory(taskText, categoryKeywords); // 키워드 포함 여부만 확인, LLM 없음
  if (category != null) return 'category_$category';
  final normalized = taskText.replaceAll(RegExp(r'\s+'), '');
  return 'text_$normalized';
}
```

- `habitId`가 있으면 습관 단위로 정확히 묶인다 (LLM 불필요, 오탐 없음, 최우선).
- 없으면 미리 정해둔 소수 카테고리(위 예시)에 키워드가 걸리는지 확인한다. 걸리면 그 카테고리로 묶여서, "설거지"에 쌓인 저항이 텍스트가 전혀 다른 "청소기 돌리기"에도 전이된다. **LLM 호출 없이 문자열 포함 여부만 확인**하므로 공짜고 결과가 항상 일관적이다.
- 카테고리에도 안 걸리면(일회성 'today' 태스크이면서 사전에 없는 활동) 정규화된 텍스트가 완전히 같을 때만 같은 그룹으로 묶인다.
- **트레이드오프**: 카테고리 사전은 미리 등록해둔 항목만 커버한다. 사전에 없는 활동(예: 처음엔 "콘텐츠 제작"이 없었음)은 다음 단계(텍스트 일치)로 떨어질 뿐 시스템이 깨지진 않는다 — 그 활동의 "비슷한 일끼리 저항 공유" 보너스만 못 받는다. 사전은 실사용하면서 자주 놓치는 게 보이면 그때그때 키워드만 추가하는 식으로 키운다 (LLM 기반 완전 자동 분류는 여전히 하지 않음 — 위에서 설명한 이유로 실익 대비 복잡도가 높다고 판단).
- **부수 효과**: 이 계산은 로컬에서 즉시 되므로, 이벤트 기록과 그룹 프로필 갱신이 **동기적으로 한 번에** 끝난다 (구 설계의 "이벤트 저장 → 비동기 LLM 분류 → 재조회 후 갱신" 2단계 파이프라인이 사라짐).

### 3.1 `TaskResistanceEvent` (SharedPreferences 키: `nyang_resistance_events`)

```jsonc
{
  "id": "evt_xxx",
  "taskId": "task_xxx",       // 원본 TaskItem.id 참조
  "taskText": "운동하기",       // 이력 보존용 스냅샷
  "groupId": "habit_abc123",  // 3.0 방식으로 기록 시점에 결정적으로 계산, 이후 변경 없음
  "date": "2026-07-15",
  "signalType": "explicit" | "implicit",
  // explicit: 채팅에서 해당 태스크 언급 중 거부신호 감지
  // implicit: 명시적 발화 없이 3일 이상 미완료 지속
  "intensity": 0.0,           // 0~1, 추후 확장용 (지금은 explicit=1.0, implicit=0.4 고정값으로 시작)
  "completedEventually": true, // 결국 완료했는지 (완료 O + 저항 O 케이스를 잡아내는 게 핵심)
  "completionOrder": 5,        // 그날 몇 번째로 완료했는지 (null이면 미완료) — done 타임스탬프만으로 계산, 추가 계측 불필요
  "totalTasksThatDay": 6
}
```

### 3.2 `ResistanceGroupProfile` (SharedPreferences 키: `nyang_resistance_groups`)

```jsonc
{
  "groupId": "habit_abc123",
  "sampleText": "운동하기",      // 화면 표시/디버깅용 예시 텍스트
  "resistanceScore": 0.72,      // 순수 신호 기반 원래 저항강도 (4장 공식, 개입 성공 여부로는 갱신 안 함 — 6장 참고)
  "confidence": 0.55,           // 누적 가중치 기반 신뢰도 (4.1 참고) — 이벤트 몇 건으로 나온 점수인지 반영
  "interventionMode": "active", // "active" | "tapering_test" | "faded" (6장 상태머신)
  "consecutiveSuccessCount": 0, // 개입 후 저항 없이 넘어간 연속 횟수
  "taperingTestConsumed": false, // tapering_test 진입 후 "개입 건너뛰기" 1회를 이미 소진했는지
  "lastUpdated": "2026-07-15",
  "eventCount": 8
}
```

**활성 그룹은 최대 5개.** 신규 후보의 초기 점수가 현재 최하위 슬롯의 `resistanceScore`를 넘어설 때만 교체한다 (밀려난 그룹의 원본 이벤트는 3.1 로그에 남아있으므로, 나중에 다시 문제되면 재계산으로 복귀 가능).

### 3.3 `PreemptiveInterventionLog` (쿨다운 + 결과관찰용, 키: `nyang_preemptive_log`)

```jsonc
{
  "id": "pi_xxx",
  "groupId": "habit_abc123",
  "taskId": "task_xxx",
  "date": "2026-07-15",
  "message": "오늘 운동하기 있는데 언제쯤 하실 생각이세요?",
  "coachId": "cat",
  "outcome": "pending" | "resolved_low_resistance" | "resolved_high_resistance" | "ignored"
}
```

### 3.4 데이터 보존 정책

원본 이벤트(3.1)가 무한정 쌓이지 않도록 삼중으로 제한한다.

1. **시간 기반 만료**: 감쇠상수 λ=0.1 기준, `exp(-0.1×t) < 0.01`이 되는 시점(t≈46일)이 지나면 점수 계산에 기여가 사실상 0이 되므로, **60일** 지난 이벤트는 자동 삭제.
2. **그룹당 개수 상한**: 그룹 하나당 최근 **12건**까지만 보관. 초과분은 가장 오래된 것부터 삭제. (이벤트가 정확히 하나의 groupId에만 속하므로, 초기 설계에서 있었던 "태그 도입 전까지 taskId로 임시 적용" 같은 어중간함 없이 처음부터 정확히 그룹 단위로 적용됨.)
3. **하루 중복 방지**: 같은 태스크에 대해 하루 안에 여러 번 거부신호가 잡혀도 이벤트는 **하루 1건만** 기록.

활성 그룹이 5개로 캡돼 있으므로, 최대 총 저장량은 5 × 12 = **60건** 수준으로 유지된다.

## 4. 저항점수 계산

```
resistScore(group) = Σ [ w_recency(event_i) × w_type(event_i) ] for event_i in group
                      ─────────────────────────────────────────
                      Σ [ w_type(event_i) ] for event_i in group

w_recency(event) = exp(-λ × daysSince(event))   // λ = 감쇠상수, 기본 0.1 (약 일주일 반감기)
w_type(event) = 1.0 if explicit else 0.4

confidence(group) = min(1, Σ[w_type(event_i)] / K)   // K = 10 (기본), 이벤트 누적 가중치가 K에 도달하면 신뢰도 100%
```

- 이벤트 1건짜리 고점수와 20건짜리 고점수를 동일 취급하지 않기 위해 `confidence`를 분리한다. 5장 개입판단에서 `confidence`가 낮으면(예: <0.3) 점수가 threshold를 넘어도 곧바로 개입하지 않고 데이터가 더 쌓일 때까지 보류한다.
- `completedEventually=true`이면서 `intensity` 높은 이벤트가 반복되는 그룹을 우선 타겟으로 삼는다.
- **LLM 호출이 전혀 없다.** groupId 계산(3.0)이 결정적이라 이벤트 기록과 동시에 점수·신뢰도까지 동기적으로 갱신된다.

**(폐기) 임베딩 기반 유사도 확장**: 초기 설계엔 "완전히 새로운 태스크도 과거 유사 태스크의 유사도로 저항을 추정"하는 4.2 확장이 있었으나, 3.0에서 설명한 이유로 이 케이스 자체를 다루지 않기로 해서 폐기했다. 필요성이 재확인되면 이 문서를 갱신해 재도입 검토.

## 5. 개입 판단 & 쿨다운

### 5.1 임계값 — 고정 매직넘버 대신 바닥값 + 오늘의 최고점 하나만

`resistanceScore ≥ 0.5`처럼 고정값 하나로 판단하면, 어떤 날은 아무도 안 걸리고 어떤 날은 여러 그룹이 동시에 걸릴 수 있다. 그렇다고 "임계값 없이 오늘의 1등을 무조건 개입"하면, 모든 그룹 점수가 낮은(진짜 아무 문제 없는) 날에도 억지로 개입하게 되는 부작용이 생긴다.

그래서 두 규칙을 같이 쓴다:
- **바닥값(FLOOR)은 유지** — 이 밑이면 그날은 아예 개입 후보가 없는 게 정상.
- **바닥값을 넘는 그룹이 여럿이면, 그중 점수가 제일 높은 것 하나만** 개입 (여러 개 동시에 찌르지 않음).

```dart
ResistanceGroupProfile? pickTodaysInterventionTarget(List<ResistanceGroupProfile> groups) {
  final candidates = groups.where((g) =>
      g.resistanceScore >= FLOOR &&              // 기본 0.5, 추후 사용자별 점수 분포 기반 상대값으로 검토
      g.confidence >= MIN_CONFIDENCE               // 기본 0.3 — 데이터 부족하면 점수 높아도 보류 (4장 참고)
  ).toList();
  if (candidates.isEmpty) return null;
  candidates.sort((a, b) => b.resistanceScore.compareTo(a.resistanceScore));
  return candidates.first;
}

bool shouldPreemptivelyIntervene(TaskItem task, ResistanceGroupProfile group) {
  if (task.done) return false;
  if (_isVacationMode()) return false;                          // 번아웃 휴식모드와 충돌 방지

  final cooldownDays = switch (group.interventionMode) {
    'faded' => 14,        // 6장 상태머신: 줄어든 상태
    _ => 3,                // active / tapering_test 기본
  };
  if (_hasRecentPreemptiveLog(group.groupId, withinDays: cooldownDays)) return false;

  // tapering_test 모드는 "개입을 한 번 건너뛰고 지켜보는" 상태이므로,
  // 조건을 통과해도 실제 발송은 스킵하고 결과만 관찰한다 (6.2 상태머신 입력용).
  if (group.interventionMode == 'tapering_test' && !group.taperingTestConsumed) {
    group.taperingTestConsumed = true; // 이번 회차는 스킵 처리했다는 표시
    return false;
  }

  return true;
}
```

기존 `_maybeOfferRest`의 쿨다운/가드레일 패턴을 그대로 재사용 (신규 로직으로 중복 구현하지 않음). FLOOR을 고정값이 아니라 사용자 본인의 최근 점수 분포 기준 상대값(예: 상위 percentile)으로 바꾸는 건 데이터가 쌓인 뒤 검토할 오픈 이슈로 남긴다 (10장).

### 5.2 전역 일일 상한 (그룹 쿨다운과 별개) — 실제 언급 확인 후 소진

그룹별 쿨다운(3일/14일)과 별개로, **하루 전체를 통틀어 선제개입은 최대 1회**로 제한한다. 오늘 날짜로 이미 `PreemptiveInterventionLog`가 하나라도 있으면 다른 그룹이 조건을 만족해도 개입하지 않는다.

**"이미 했다"의 판정은 코치의 실제 응답 텍스트에 해당 태스크가 언급됐는지 확인한 뒤에만 이뤄진다** (지침을 프롬프트에 주입한 시점이 아님). 이렇게 나눈 이유: 하루의 첫 대화 턴에서 즉시 소진 처리하면, 하필 그 턴이 대화 흐름상 어색해서 코치가 화제를 안 꺼냈을 때 이후 더 자연스러운 타이밍(예: 오후에 사용자가 피곤하다고 말하는 순간)이 와도 재시도할 기회가 없어진다. 그래서:

1. `findPreemptiveInterventionTarget()`: 로그를 남기지 않는 순수 조회. 매 턴 호출되어 프롬프트에 지침을 주입할지만 결정한다.
2. `confirmPreemptiveIntervention()`: 코치 응답이 나온 뒤, 응답 텍스트에 대상 태스크가 실제로 언급됐는지 정규화 부분일치로 확인하고, 언급됐을 때만 로그를 남겨 그날의 기회를 소진시킨다.

이러면 하루 안에서 여러 턴에 걸쳐 자연스러운 타이밍을 계속 노릴 수 있고, 끝내 타이밍이 안 나오면 정말로 그날은 개입 없이 넘어간다 — 이게 "매일 기계적으로 개입" 대신 원했던 자연스러운 빈도다.

## 6. 결과 관찰 & 가중치 갱신 (폐루프)

### 6.1 설계 원칙 — "개입 성공"과 "저항 소멸"을 혼동하지 않는다

초안에서는 개입이 성공하면 `resistanceScore`를 바로 낮췄는데, 이건 인과관계 오류다. 개입이 성공했다는 건 **개입이 효과가 있었다**는 뜻이지 **원래 저항이 사라졌다**는 뜻이 아니다 (혈압약 먹고 정상 수치 나왔다고 약을 끊으면 안 되는 것과 같음). 그래서 두 가지를 분리한다:

- `resistanceScore` / `confidence` (3.2): **순수 신호로만 갱신**. 개입 없이 발생한 explicit/implicit 이벤트만 반영 (4장 공식 그대로). 개입 성공/실패로는 절대 직접 건드리지 않음.
- `interventionMode` + `consecutiveSuccessCount`: 개입을 계속할지, 줄여볼지를 별도로 판단하는 상태머신.

**저항 지속 여부 판정**: "그날 이 태스크에 대한 저항 이벤트가 하나라도 있었는가"로 판정한다. 상태머신이 알고 싶은 건 "이 그룹이 오늘도 여전히 말썽이었는가"이지 "우리 개입이 인과적으로 효과가 있었는가"가 아니므로, 개입 전/후 순서를 정밀하게 구분할 필요는 없다 (검토 후 확정).

### 6.2 상태머신

```
active (기본값, 매번 개입)
  └─ 개입 후 무저항 완료 3회 연속 → tapering_test로 전환, consecutiveSuccessCount 리셋

tapering_test (한 번 개입을 건너뛰고 지켜봄)
  ├─ 저항 없이 잘 넘어감 → faded로 전환
  └─ 저항 다시 발생 → active로 즉시 복귀, consecutiveSuccessCount = 0

faded (개입 빈도 낮춤 — 예: 쿨다운을 3일→14일로 늘림)
  └─ 저항 신호(raw event)가 다시 threshold 넘으면 → active로 즉시 복귀
```

**참고**: `faded` 상태에서도 `resistanceScore`는 계속 raw 신호로 갱신되니, 개입을 줄인 사이에 실제로 다시 저항이 쌓이면 그 신호로 재감지된다. 상태전환과 점수갱신이 서로 다른 데이터소스를 쓰기 때문에 요요 문제가 안 생긴다.

## 7. 기술 스택 결정 사항

| 구성요소 | 최종 선택 | 이유 |
|---|---|---|
| 태스크 그룹핑 | `habitId` 또는 정규화 텍스트 기반 결정적 계산 | LLM 불필요, 오탐 없음, 실제 반복 저항의 대다수(습관/일정)를 정확히 커버 |
| 저장소 | SharedPreferences (기존 `nyang_history` 패턴 재사용) | 신규 인프라 불필요 |
| 계산 위치 | **클라이언트(Flutter) 내부** 1차, 추후 서버 이전 검토 | MVP는 속도 우선. 단, 영업비밀 보호 관점에서 정식 버전은 서버 이전 권장 (8장 참고) |

### 7.1 LLM 호출 — 이 기능은 추가 LLM 호출이 전혀 없다

`TaskResistanceService`(점수계산·그룹핑·개입판단·상태머신 전부)는 **순수 계산/코드 로직**이며 어디에도 LLM을 호출하지 않는다. 초기 설계에 있던 "태스크 → 태그 분류" LLM 호출(3.0 참고)을 걷어낸 뒤로는 이 서비스에 `cloud_functions` 의존성 자체가 없다.

유일하게 LLM과 맞닿는 지점은 **개입 문구 전달 방식**뿐이다:
- 초기엔 하드코딩 템플릿 계획 → 대화 맥락에 자연스럽게 연결되는 화제 제시가 필요해져 **기존 코칭 LLM 호출(`chatProxy`)의 시스템 프롬프트에 지침 블록을 주입**하는 방식으로 전환.
- `_buildContextString()`이 일반 대화 턴과 인사말 양쪽에 이미 쓰이는 공용 함수라, 여기 한 곳에 지침을 추가하는 것만으로 "채팅을 열었을 때"와 "일반 대화 중" 두 트리거를 모두 커버함.
- **새 LLM 호출이 추가되는 게 아니라, 이미 매 턴 발생하는 코칭 호출의 프롬프트에 텍스트가 조금 늘어나는 것뿐**이라 이 기능으로 인한 API 비용 증가는 사실상 없다.

## 8. 영업비밀 보호 고려사항

- MVP는 클라이언트에서 계산해도 되지만, **정식 배포 시 저항점수 계산 로직(가중치 공식, 임계값, 갱신 규칙)은 서버(Cloud Function 등)로 이전**할 것. 클라이언트 바인너리 리버스엔지니어링(Flutter AOT 스냅샷 분석 도구, 예: reFlutter)으로 문자열 상수·로직이 노출되는 걸 방지.
- **네트워크 트래픽 노출 별도 주의**: 계산 로직을 서버로 옮겨도, 그 서버가 LLM에 보내는 프롬프트 자체가 "왜 이렇게 판단했는지"를 너무 자세히 담고 있으면 트래픽 가로채기(mitmproxy 등)로 새어나갈 수 있다. 서버 내부 로직(임계값·상태머신 판단)과 LLM에 넘기는 프롬프트(결과값만 전달, 판단 근거 노출 최소화)를 분리해서 설계할 것.
- 서버 이전 시에도 원본 이벤트 데이터까지 통째로 Firestore에 상시 저장하면 읽기/쓰기 비용이 발생한다. 데이터는 클라이언트(SharedPreferences)에 그대로 두고, **판단 로직만 스테이트리스 서버 함수로 호출**(클라이언트가 로컬 데이터를 요청 시 함께 전달, 서버는 계산 결과만 반환)하는 방식이 저장 비용 없이 로직 은닉 목적을 달성하는 절충안.
- 소스 저장소에서 이 문서와 관련 커밋은 별도 접근 제한 고려 (사내 NDA 대상자만).
- 정확한 임계값 상수(0.5, 0.1, 10.0 등)는 문서/커밋 메시지에 노출하지 말고 설정값으로 분리해서 관리.

## 10. 오픈 이슈

- 페르소나 로테이션(효과 낮으면 다른 코치로 시도)은 이번 설계에 포함 안 함 — 별도 설계 필요할지 다음에 논의.
- 5.1의 FLOOR(현재 고정값 0.5)를 사용자별 점수 분포 기반 상대값(예: 상위 percentile)으로 바꿀지 — 데이터 충분히 쌓인 뒤 검토.
- "완전히 새로운 일회성 태스크가 과거 다른 태스크와 개념적으로 비슷해서 묶이는" 케이스는 사전 정의 카테고리(4개 시드)로 부분 커버됨. 사전 밖 활동은 여전히 미지원 — 실사용하며 자주 놓치는 카테고리가 보이면 키워드만 추가.
- 8장에서 언급한 서버 이전(계산 로직 + 프롬프트 분리)은 아직 미착수 — 이 리포지토리 밖의 Cloud Functions 백엔드 작업 필요.
